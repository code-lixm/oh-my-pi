**任务通过逐字内容字符串引用，NEVER 是自动生成的 ID——不存在“任务-1”/“任务-N”。请在 `task` 字段中传入内容文本。**

每次完成时，最早仍未关闭的任务（按阶段顺序）都会自动提升为 `in_progress`。乱序完成任务可能会把这个指针**移回**更早的阶段——这是预期行为；已完成的任务绝不会被回退。

## 操作

|`op`|必填字段|效果|
|---|---|---|
|`init`|`list: [{phase, items: string[]}]`|初始化完整列表（替换现有列表）|
|`init`|`items: string[]`|扁平单阶段初始化|
|`start`|`task`|标记为进行中|
|`done`|`task` 或 `phase`|标记为已完成|
|`drop`|`task` 或 `phase`|标记为已放弃|
|`rm`|`task` 或 `phase`（可选）|移除任务或阶段中的任务；两者都省略则清空列表|
|`append`|`phase`、`items: string[]`|向 `phase` 追加任务；按需惰性创建阶段|
|`view`|—|只读：回显列表，不做修改|

## 结构
- **任务内容**：5–10 个词；说明做什么，不说明怎么做。唯一标识符。
- **阶段名称**：简短名词短语（例如 `Foundation`、`Auth`、`Verification`）。唯一标识符。NEVER 前缀 `1.`、`A)`、`Phase 1:`。

## 规则
- 完成任务后立即将其标记为已完成。
- 按顺序完成各阶段。
- 被阻塞了？向活动阶段 `append` 一个任务以解除阻塞，或者 `drop`。
- 一旦引入，保持 `task`/`phase` 字符串稳定不变。
- 丢失了精确的任务文本？`view` 会回显列表——NEVER 凭记忆猜测；不匹配的 `task` 字符串会导致错误。

## 何时创建列表
- 任务需要 3 个以上不同步骤
- 用户明确请求创建一个
- 用户提供了一组任务
- 任务进行中收到新指令——先记录再继续

<critical>
用户交给你一个多步骤计划——分阶段待办、numbered/bulleted 清单，或 “N bugs/items/tasks”：
- 在开始处理前，你 MUST 用每一项各自作为单独任务来 `init` 这个列表。
- 全部枚举出来；NEVER 不要概括成更少的任务，不要只举“重要的那些”作为示例，不要遗漏项目，也不要靠记忆跟踪其余内容。
</critical>
